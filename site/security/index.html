Security model

Deliberate controls around what can run and when

Secure defaults

  • Exact alias matching instead of raw host entry.
  • Named profiles instead of free-form shell by default.
  • Publickey-only non-interactive SSH in the secure path.
  • Rendered command and result written to the audit log.

Approval gating

A server or profile can require approval before execution. That same approval gate is required for unrestricted sessions. 

agent-ssh exec \
  --server prod-web-1 \
  --profile logs \
  --arg service=nginx \
  --arg "since=5 min ago" \
  --approval CAB-1234

Unrestricted sessions are explicit, not implied

Arbitrary commands are only available when all of these are true:

  • The server has allow_unrestricted_sessions = true.
  • The server also has requires_approval = true.
  • The user settings in the agent-facing layer allow arbitrary commands.

That means “run anything” is still policy-bound. It is not silently enabled by using the CLI or agent SDK.

Legacy password compatibility

Legacy password auth exists only as a migration path. Password material must not appear in TOML, CLI args, or audit output, and persistent sessions are not supported in that mode.

Root login is also strongly discouraged and only allowed as an explicit break-glass exception with acknowledgment in config.
